## 1. Goal

Today, ceiling and allocation setup is effectively **single-fund** — there is exactly one
`budget_ceilings` row per `(office, fiscal_year)` and one `division_allocations` row per
`(division, fiscal_year)`, with no fund-source dimension. In practice that single pot behaves as
the **General Fund** budget.

This feature extends ceiling and allocation setup to **all active fund sources** (driven by the
existing `funding_sources` config table), so each fund source has its own office ceiling, its own
per-division allocation, and its own independent WFP over-allocation tracking.

The fund sources currently in use (final list follows the `funding_sources` config — the setup UI
must read it dynamically, not hardcode this list):

- General Fund *(the existing ceiling/allocation behaves as this today)*
- 20% Development Fund
- 5% GAD Fund
- 5% Local Disaster Risk Reduction and Management Fund
- 1% Provincial Council for the Protection of Children Fund
- 1% Senior Citizen / Persons with Disability Fund
- Special Education Fund

## 2. Decisions (settled)

| # | Decision |
|---|---|
| D1 | Ceiling/allocation setup is driven by the **active** `funding_sources` config rows — no hardcoded fund-source list anywhere. Inactive sources are excluded from the Allocation page and the WFP entry dropdown. |
| D2 | The Allocation page's "Ceiling & Division Allocation" tab shows a ceiling + a division-allocation table (+ stacked bar) **per active fund source**. All active fund sources are displayed. |
| D3 | **AIP-activity-total budget check stays aggregate** — it keeps summing all expenditures for an AIP activity (across fund sources) against the one `AipActivity.Total × 1000`. AIP data carries no per-fund breakdown, so there is nothing to split it against. |
| D4 | **Division-allocation check becomes fund-scoped** — each expenditure is validated against *its own fund source's* division allocation and its own fund-scoped ledger balance. This is the core behavioural change. |
| D5 | WFP entry **fund source defaulting**: when the AIP activity has a single, cleanly-resolvable fund source → default to it. When the activity's fund source is **blank/none** → default to **General Fund** (with a hint). When the activity's fund source is **ambiguous/multi-valued** → leave **unselected**, user must choose (option-b), but keep a swappable fallback constant so "default to GF" (option-a) can be turned on later without a refactor. |
| D6 | WFP entry **allocation bars**: the set of fund-source bars shown for the selected activity is General Fund (**always**) ∪ the activity's own **resolved AIP fund source** if it is a single non-GF source (e.g. an activity whose AIP fund source is GAD shows **both GF and GAD** bars, even before any expenditure uses GAD) ∪ any fund source actually **used by an expenditure** on the activity. |
| D7 | **Setup gate**: only the **General Fund** ceiling + allocation is mandatory to unlock WFP entry (`GetSetupStatusAsync`). Other fund sources are optional and only enforced (via the block-on-save check) once an expenditure actually selects that fund source. |
| D8 | **Backfill**: existing `budget_ceilings` and `division_allocations` rows are backfilled to `funding_source_id = General Fund`. Existing `wfp_division_allocation_ledger` rows are backfilled to General Fund. Existing `WfpExpenditure`/`WfpExpenditureLine` rows with `FundingSourceId = null` are backfilled to General Fund explicitly (set the FK), so the fund-scoped ledger can attribute historical usage correctly. |
| D9 | **Fund-source note is a live hint only** (not persisted). Two cases: (a) shown when the fund source was auto-defaulted to General Fund because the activity had no fund source assigned; (b) shown when the user manually selects a fund source **different** from the one AIP assigned to the activity (only when AIP did assign one). |
| D10 | **Out of scope:** the Purchase Requests module — `PurchaseRequest.Fund` (free-text) and the hardcoded `-GF-` segment in PR-number generation ([`PurchaseRequestService.cs:464`](../../backend/PPDO.Application/Services/PurchaseRequestService.cs)) are **not** touched by this feature. |

## 3. Data model changes

All three allocation tables gain a `funding_source_id` FK and a widened unique key. New columns are
**snake_case** (these are v1.2+ snake_case tables already).

### 3.1 `budget_ceilings`
- Add `funding_source_id INT NOT NULL` (FK → `funding_sources.id`, `OnDelete: Restrict`).
- Drop unique `IX_budget_ceilings_office_fy` `(office_id, fiscal_year)`.
- Add unique `IX_budget_ceilings_office_fy_fund` `(office_id, fiscal_year, funding_source_id)`.

### 3.2 `division_allocations`
- Add `funding_source_id INT NOT NULL` (FK → `funding_sources.id`, `OnDelete: Restrict`).
- Drop unique `(division_id, fiscal_year)`; add unique `(division_id, fiscal_year, funding_source_id)`.

### 3.3 `wfp_division_allocation_ledger`
- Add `funding_source_id INT NOT NULL` (FK → `funding_sources.id`, `OnDelete: Restrict`).
- Change the logical key from `(division_id, fiscal_year, wfp_record_id)` to
  `(division_id, fiscal_year, funding_source_id, wfp_record_id)`. A single WFP record can now post
  **one ledger row per fund source** it spends from (rather than one row total), because a record's
  expenditures may draw from multiple fund allocations.

### 3.4 Backfill (same migration, data step — see D8)
- `UPDATE budget_ceilings SET funding_source_id = @gfId` for all existing rows.
- `UPDATE division_allocations SET funding_source_id = @gfId` for all existing rows.
- `UPDATE wfp_division_allocation_ledger SET funding_source_id = @gfId` for all existing rows.
- `UPDATE wfp_expenditures SET funding_source_id = @gfId, funding_source_snapshot = 'GF',
  funding_source_name_snapshot = 'General Fund' WHERE funding_source_id IS NULL` (and the legacy
  `wfp_expenditure_lines` equivalent). Resolve `@gfId` from `funding_sources` by `Code = 'GF'`
  (fail the migration loudly if the GF row is missing — it must be seeded first).

> **Migration safety:** adding a NOT NULL column to populated tables requires the add-nullable →
> backfill → alter-to-NOT-NULL → add-unique-index sequence within the one migration. Keep the
> ledger key change and the expenditure backfill in the same migration so no intermediate state
> has null fund sources.

## 4. Backend behaviour changes

### 4.1 `AllocationService`
- `GetCeilingAsync` / `UpsertCeilingAsync` gain a `fundingSourceId` parameter; find/upsert is keyed
  on `(office, FY, fundingSource)`. Add a `GetCeilingsAsync(officeId, fiscalYear)` returning **all
  fund sources' ceilings** for the page load.
- `GetAllocationsAsync` / `UpsertAllocationsAsync` gain `fundingSourceId`; the Σ-allocations ≤
  ceiling guard (Guard 2) is now **per fund source** (sum only that fund's division amounts vs.
  that fund's ceiling). Guard 1 (ceiling must exist) is per fund source too.
- `GetSetupStatusAsync` (D7): `HasCeiling`/`HasAllocation` check the **General Fund** rows only —
  other fund sources do not block the gate.
- `GetSetupOverviewAsync`: "fully set up" still keys on General Fund ceiling + GF allocation +
  program assignment (keep the dashboard semantics stable; other funds are optional).
- DTOs (`BudgetCeilingDto`, `DivisionAllocationDto`, `UpsertCeilingDto`, `UpsertDivisionAllocationDto`,
  `UpsertAllocationsDto`) gain a `FundingSourceId` (and the read DTOs a `FundingSourceCode`/`Name`
  for display).

### 4.2 `WfpCeilingService`
- **AIP check — unchanged** (D3): keep summing across all fund sources vs. `AipActivity.Total × 1000`.
- **Division allocation check — fund-scoped** (D4): the check must be parameterised by the
  expenditure's `FundingSourceId`.
  - `GetDivisionAllocationAsync(division, fiscalYear, fundingSourceId)` reads that fund's allocation.
  - Ledger sums (`SumUsedAmountAsync`) become per fund source — add `fundingSourceId` to the ledger
    repo signature and the ledger key.
  - `UpsertLedgerForActivityAsync` must upsert **one ledger row per (division, FY, fund source,
    wfp record)** — group the record's expenditures by `FundingSourceId` and post a row per group.
  - When an expenditure has no fund source at save time, it is treated as General Fund (matches the
    backfill and the entry-form default) — resolve to the GF `funding_sources` row.
- `GetStatusAsync` / `WfpCeilingStatusDto`: return a **per-fund-source breakdown** the entry page
  can render (see 4.3). Keep the aggregate AIP budget/used at the top level; nest a list of
  `{ fundingSourceId, code, name, allocation, remaining }` for each fund source relevant to the
  activity (always include General Fund).
- `ValidateRecordForFinalizeAsync`: the division-allocation backstop becomes per fund source too.

### 4.3 Ceiling-status endpoint (`GET /api/budget-planning/wfp/ceilings`)
- Extend `WfpCeilingStatusDto` with the per-fund-source list described above. No route change; the
  entry page's `getCeilingStatus` consumes the richer payload.

## 5. Frontend behaviour changes

### 5.1 Allocation page — Tab 1 "Ceiling & Division Allocation"
- Load active fund sources (`listFundingSources({ active: true })`) plus all fund sources' ceilings
  and allocations for the office+FY.
- Render a **per-fund-source section** (General Fund first, then the rest by config order). Each
  section has: the fund source name/color header, a ceiling `MoneyInput` + "Set Ceiling", the
  division-allocation table + stacked `AllocationBar`, and the live "Allocated X of Y · Remaining"
  totals — i.e. the current single-fund UI, repeated per fund source.
- Recommended layout: **General Fund expanded by default; other fund sources as collapsible
  cards** showing a one-line summary (ceiling set / not set, Σ allocated) until expanded — avoids a
  wall of 7 full tables. (See the mockup.)
- Reuse `MoneyInput`, `AllocationBar`, `DIVISION_COLORS`, flat design, green accent. The per-fund
  ceiling/allocation save calls carry `fundingSourceId`.
- Tab 2 "PPA → Division" is **unchanged** (program→division assignment is not fund-scoped).

### 5.2 WFP entry page
- **Fund source default (D5):** update `resolveDefaultFundingSourceId`
  ([`wfp/entry/page.tsx:105-120`](../../frontend/src/app/(portal)/budget-planning/wfp/entry/page.tsx)):
  - single resolvable AIP fund source → that source (current behaviour);
  - blank/none → **General Fund** (new);
  - ambiguous (comma/slash) → `null` / unselected (current behaviour, option-b). Put the
    "blank → GF" fallback behind a single named constant/flag (e.g. `DEFAULT_FUND_SOURCE_CODE =
    "GF"` + `FALLBACK_AMBIGUOUS_TO_DEFAULT = false`) so option-a can be enabled later.
- **Hints (D9):** in the expenditure popup's Fund Source field, show a small inline note when
  (a) the value was auto-defaulted to General Fund because the activity had no fund source, or
  (b) the selected fund source differs from the activity's AIP-assigned fund source. Hints are
  display-only; nothing is persisted.
- **Allocation bars (D6):** the sticky ceiling context header shows the General Fund allocation bar
  always, plus the activity's own resolved AIP fund-source bar when that is a single non-GF source
  (e.g. GAD → show GF + GAD), plus a bar for any other fund source that has at least one expenditure
  on the selected activity. Drive these from the per-fund-source `WfpCeilingStatusDto` list.
- The live debounced pre-save check (`useCeilingStatus`) now compares the pending amount against the
  **selected fund source's** allocation remaining, matching the server's fund-scoped block-on-save.
