## 1. User Profile Page — Change Password (RAL-88)

### 1A. Overview

A self-service profile page at `/account` accessible to all authenticated users. The initial release ships one tab (Security) with a password change form. The page is designed to accommodate future profile fields (display name, email, notification preferences) as additional tabs.

### 1B. Backend

**No migration required** — the password column already exists on the `users` table.

**New service method on `IUserService` / `UserService`:**

```csharp
Task<ServiceResult> ChangePasswordAsync(CallerContext caller, ChangePasswordDto dto, CancellationToken ct);
```

`ChangePasswordDto`:
```csharp
public sealed record ChangePasswordDto(
    string CurrentPassword,
    string NewPassword,
    string ConfirmPassword);
```

**Validation order:**
1. Load user by `caller.UserId` — `NotFound` if missing
2. Verify `CurrentPassword` against stored hash — `Validation("Current password is incorrect")`
3. `NewPassword == ConfirmPassword` — `Validation("Passwords do not match")`
4. Password policy: min 8 chars, ≥ 1 uppercase letter, ≥ 1 digit — `Validation(...)` on failure
5. Hash `NewPassword` using the existing password hasher, save, return `Success()`

**New API endpoint:**

| Method | Route | Auth | Response |
|--------|-------|------|----------|
| PUT | `/api/users/me/password` | JWT (any role) | 204 No Content on success; `ApiResponse<T>` envelope on error |

### 1C. Frontend

**Route:** `src/app/(portal)/account/page.tsx`

**Sidebar:** "My Account" item visible to all authenticated users, `UserCircle` Lucide icon, positioned near the bottom of the sidebar (above logout).

**Page layout:**
```
My Account
[Security tab]

  Change Password
  ┌─────────────────────────────────────────────┐
  │ Current Password  [                        ] │
  │ New Password      [                        ] │
  │ Confirm Password  [                        ] │
  │                              [Save Changes]  │
  └─────────────────────────────────────────────┘
  [inline error message here if API returns error]
```

**Behaviour:**
- Client-side validation fires before submit (new ≠ current; confirm matches; policy check)
- Inline red error below form for API errors (wrong current password, etc.)
- On success: clear all three fields + show success toast "Password updated successfully."
- Save button disabled until all fields non-empty; shows loading spinner while in-flight

**API helper:** `lib/account.ts` → `changePassword(dto): Promise<void>`

### 1D. TDD

Test `ChangePasswordAsync`:
- Correct current password + valid new password → hash updated
- Wrong current password → `ServiceResult.Validation`
- New ≠ confirm → `ServiceResult.Validation`
- Policy failure → `ServiceResult.Validation`
- Missing user → `ServiceResult.NotFound`

## 2. Calendar Event Approval Workflow

### 2A. Overview

The existing dashboard calendar allows any authenticated user to create Office (shared) or Personal (private) events. This feature adds an approval step so non-admin Office event submissions are queued for review before becoming visible to all users.

**Approval rules:**

| Creator | Event type | Result |
|---------|-----------|--------|
| Admin / SuperAdmin | Office | Immediately `Approved` — visible to all |
| Admin / SuperAdmin | Personal | Immediately visible (private) — unchanged |
| Staff / Observer / Office user | Office | Created as `Pending` — not visible to others until reviewed |
| Staff / Observer / Office user | Personal | Immediately visible (private) — no approval needed |

### 2B. DB Schema — RAL-82

**Migration: `AddCalendarEventApproval`**

ALTER `calendar_events` table — add four nullable columns (no data loss):

| Column | Type | Default | Purpose |
|--------|------|---------|---------|
| `status` | `int NOT NULL` | `1` (Approved) | 0=Pending, 1=Approved, 2=Rejected — DEFAULT 1 keeps all existing rows as Approved |
| `reviewed_by_id` | `Guid? FK → Users (Restrict)` | NULL | Admin who reviewed |
| `reviewed_at` | `DateTime?` | NULL | When reviewed |
| `rejection_reason` | `nvarchar(500)?` | NULL | Shown to creator on rejection |

Add index on `status` for the pending-events query.

**Domain changes:**
- ADD `CalendarEventStatus` enum: `Pending = 0, Approved = 1, Rejected = 2`
- UPDATE `CalendarEvent` entity: add `Status`, `ReviewedById`, `ReviewedAt`, `RejectionReason`, `ReviewedBy` nav prop
- UPDATE `CalendarEventConfiguration.cs`: map new columns, FK constraint, index

**Branch:** `feature/v1.1.1-ral-82-calendar-event-approval-db`

### 2C. Backend API — RAL-84 (blocked by RAL-82)

**Updated `CreateEventAsync`:**
- Office event by non-admin → `Status = Pending`
- Office event by admin → `Status = Approved`
- Personal events → no status change

**Updated `GetEventsAsync`:**
- Return Approved Office events to all users (existing behaviour)
- ALSO return the caller's own Pending and Rejected Office events
- Another user's Pending events are NOT returned

**New service methods:**
- `GetPendingEventsAsync(caller, ct)` — Admin/SuperAdmin only; all Pending Office events ordered by `CreatedAt ASC`
- `ReviewEventAsync(caller, id, dto, ct)` — Approve (sets Approved, clears reason) or Reject (sets Rejected, requires reason string); sets `ReviewedById`, `ReviewedAt`
- `DeleteEventAsync(caller, id, ct)` — creator can delete own; admin can delete any

**New DTOs:** `ReviewCalendarEventDto(bool Approved, string? RejectionReason)`, `PendingCalendarEventDto`; update `CalendarEventDto` to include `Status` + `RejectionReason`

**New endpoints:**

| Method | Route | Auth |
|--------|-------|------|
| GET | `/api/dashboard/events/pending` | JWT, Admin/SuperAdmin |
| PUT | `/api/dashboard/events/{id}/review` | JWT, Admin/SuperAdmin |
| DELETE | `/api/dashboard/events/{id}` | JWT (creator or admin) |

**Branch:** `feature/v1.1.1-ral-84-calendar-approval-backend`

### 2D. Frontend — RAL-85 (blocked by RAL-84)

**Event color map (updated):**

| State | Color |

## 3. Public Announcements

### 3A. Overview

Admin/SuperAdmin author rich-text announcements in the portal. Published announcements appear on the public landing page, which already has a placeholder section.

**Status workflow:**
```
Draft → Published → Archived
        ↑ (can unpublish back to Draft)
```
- Deleting a Published announcement is blocked — must archive first.
- Hard delete allowed for Draft and Archived.

**Access:** Admin and SuperAdmin only — no new permission flag.

**Rich text:** TipTap editor (React, MIT). Toolbar: Bold · Italic · Underline · H2 · H3 · Bullet list · Ordered list · Text color · Font family · Clear formatting.

**XSS protection:**
- Server-side: `Ganss.Xss` HtmlSanitizer (NuGet) on every create/update
- Client-side (public page render): `DOMPurify` on `dangerouslySetInnerHTML`

### 3B. DB + API — RAL-83

**Migration: `AddAnnouncements`**

New `announcements` table:

```sql
id              UNIQUEIDENTIFIER PK
title           NVARCHAR(200) NOT NULL
content         NVARCHAR(MAX) NOT NULL   -- sanitized HTML
status          INT NOT NULL DEFAULT 0   -- 0=Draft, 1=Published, 2=Archived
published_at    DATETIME2 NULL           -- set once on first Publish
created_by_id   UNIQUEIDENTIFIER FK → users (Restrict)
created_at      DATETIME2 NOT NULL
updated_at      DATETIME2 NOT NULL
```

Indexes: `status`, `published_at DESC`.

**Domain:** `AnnouncementStatus` enum (Draft=0, Published=1, Archived=2); `Announcement` entity; `AnnouncementConfiguration.cs`.

**Application service:** `IAnnouncementService` / `AnnouncementService`

| Method | Access | Notes |
|--------|--------|-------|
| `GetPublishedAsync()` | Public | Published only, `PublishedAt DESC` |
| `GetAllAsync(caller)` | Admin/SuperAdmin | All statuses |
| `CreateAsync(caller, dto)` | Admin/SuperAdmin | Creates as Draft; sanitize content |
| `UpdateAsync(caller, id, dto)` | Admin/SuperAdmin | Editable in Draft or Published; re-sanitize |
| `PublishAsync(caller, id)` | Admin/SuperAdmin | Sets Published, populates `PublishedAt` once |
| `UnpublishAsync(caller, id)` | Admin/SuperAdmin | Published → Draft |
| `ArchiveAsync(caller, id)` | Admin/SuperAdmin | Published → Archived |
| `DeleteAsync(caller, id)` | Admin/SuperAdmin | Hard delete Draft/Archived; Conflict if Published |

**Endpoints (`AnnouncementFunctions.cs`):**

| Method | Route | Auth |
|--------|-------|------|
| GET | `/api/announcements` | **Public** (no JWT check) |
| GET | `/api/announcements/manage` | JWT, Admin/SuperAdmin |
| POST | `/api/announcements` | JWT, Admin/SuperAdmin |
| PUT | `/api/announcements/{id}` | JWT, Admin/SuperAdmin |
| PUT | `/api/announcements/{id}/publish` | JWT, Admin/SuperAdmin |
| PUT | `/api/announcements/{id}/unpublish` | JWT, Admin/SuperAdmin |
| PUT | `/api/announcements/{id}/archive` | JWT, Admin/SuperAdmin |
| DELETE | `/api/announcements/{id}` | JWT, Admin/SuperAdmin |

Public GET returns raw JSON array. All others use `ApiResponse<T>` envelope.

**NuGet:** `Ganss.Xss`

**TDD:** CreateAsync sanitizes HTML; PublishAsync sets status + PublishedAt (not updated on re-publish); UnpublishAsync; ArchiveAsync; DeleteAsync blocks Published; non-admin → Forbidden.

**Branch:** `feature/v1.1.1-ral-83-announcements-api`
